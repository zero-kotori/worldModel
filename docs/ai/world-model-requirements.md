# 世界模型工具需求文档

## 1. 背景

当前项目希望实现一个用于记录、跟踪和更新个人世界模型的私有工具。它不是公开预测市场，也不是普通笔记软件，而是一个面向个人认知管理的系统：用户可以记录信念、为信念维护一组假设，持续收集相关观察和证据，并在证据被确认后，根据似然模型和贝叶斯规则更新假设概率，从而调整信念强度。

该工具需要接入现有个人网站 `../myWeb`，但不应把世界模型的数据混入个人网站的内容数据库。第一版应以独立应用的形式运行，由 `myWeb` 的管理员后台提供同域受保护入口。

## 2. 产品目标

- 建立可编辑的预测表，用于记录一组长期信念和具体假设。
- 为每条信念维护对应的假设库，支持可结算预测和长期趋势判断。
- 建立观察池，用于承接手动录入或外部采集到的原始信息。
- 建立证据库，只让已确认的观察参与概率更新。
- 根据证据、假设和似然模型生成更新预览，并应用贝叶斯更新。
- 记录每一次概率变化的原因、输入、模型输出和更新前后状态，支持审计和回滚。
- 支持从网页、RSS、搜索、社交平台、GitHub、Hugging Face、新闻事件流和预测市场拉取候选观察。
- v1 使用 LLM API 作为主似然评分器，轻量可解释模型作为 fallback 和 sanity check，外部深度模型适配器作为后续增强。
- 支持后续把训练任务迁移到用户本地电脑运行，再把模型产物导入服务器。

## 3. 使用对象

第一版仅面向站点管理员本人使用。

系统不提供公开访问页面，不提供访客交互，不发布投资建议或预测结论。

## 4. 第一版信念范围

第一版只覆盖以下五类信念：

- AI 技术趋势判断。
- 投资领域判断。
- 其他技术趋势判断。
- 职业方向判断。
- 信息源可靠性判断。

后续可以扩展更多分类，但第一版不需要做无限制分类体系。

## 5. 核心概念

### 5.1 信念

信念是一个判断主题或预测主题的容器。一个信念下可以有多条假设。

信念需要包含：

- 标题。
- 分类。
- 描述。
- 概率结构。
- 状态。
- 创建和更新时间。

信念支持两种概率结构：

- 互斥完备：同一信念下的假设概率总和归一化为 100%。
- 相互独立：同一信念下每条假设独立更新，各自拥有 0 到 100% 的概率。

### 5.2 假设

假设是实际承载概率的单元。

每条假设需要包含：

- 命题。
- 所属信念。
- 时间范围。
- 到期条件。
- 初始概率。
- 当前概率。
- 信念强度。
- 状态。
- 结算结果。
- 备注。

假设既可以是有明确截止日期的预测，也可以是长期趋势判断。

### 5.3 观察

观察是系统从手动输入或外部来源获得的原始信息。

观察进入系统后默认只进入观察池，不直接触发贝叶斯更新。

观察需要包含：

- 标题。
- 正文。
- 原始链接。
- 作者或来源。
- 发布时间。
- 抓取时间。
- 来源可信度。
- 去重信息。
- 当前处理状态。

### 5.4 证据

证据是经过确认的观察。

证据可以由用户人工确认，也可以由满足阈值的自动确认策略确认。自动确认必须可审计、可回滚。

一条证据可以同时关联多个假设。每个证据和假设的关联都需要单独记录方向、相关性、似然、置信度和解释。

### 5.5 似然模型

似然模型用于估计 `P(证据 | 假设)` 或等价的似然比。

v1 的似然估计采用 LLM-first 架构：

- LLM API 是主评分器。系统需要把证据、假设、信念上下文、来源可信度和时间信息传给 LLM，并要求返回结构化评分。
- 可解释轻量模型作为 fallback 和 sanity check。当 LLM 不可用、低置信度、输出无效或成本需要控制时，轻量模型可以给出保守估计或提示人工复核。
- 外部深度模型适配器作为后续增强。v1 只保留接口，不作为闭环必需依赖。

LLM 主评分器必须输出结构化结果，包括方向、相关性、似然比、置信度、解释和是否需要人工复核。系统根据 LLM 输出、轻量模型校验结果和安全阈值决定自动应用或进入待审。

### 5.6 更新事件

每次贝叶斯更新都必须生成更新事件。

更新事件需要保存：

- 更新前的概率快照。
- 更新后的概率快照。
- 使用的证据。
- 参与的假设。
- 似然模型输出。
- 模型版本。
- 置信度。
- 更新理由。
- 更新时间。
- 是否已回滚。

## 6. 主要工作流

### 6.1 创建信念和假设

用户进入世界模型后台，创建一个信念，选择分类和概率结构，然后创建一组相关假设。

系统需要校验：

- 信念标题不能为空。
- 假设命题不能为空。
- 初始概率必须在 0 到 1 之间。
- 互斥完备模式下，假设概率需要能够归一化。

### 6.2 手动录入观察

用户可以手动录入观察，包括标题、正文、链接、来源和相关备注。

手动观察进入观察池，状态为待处理。

### 6.3 外部来源采集

用户可以配置观察来源，包括：

- RSS。
- 普通网页。
- 搜索结果。
- GitHub 趋势或事件。
- Hugging Face 模型或数据集。
- GDELT 或新闻事件流。
- 预测市场 API。
- 社交平台适配器。

系统按来源配置拉取候选观察，并记录每次采集的运行状态、抓取数量、去重数量和错误信息。

### 6.4 去重和未知证据处理

观察进入观察池后，系统需要进行重复检查。

去重依据包括：

- 原始 URL。
- 规范化文本 hash。
- 语义近似 key。
- 时间窗口。
- 来源聚类。

疑似重复观察不应直接丢弃，而应标记为重复候选，允许用户查看和确认。

无法明确关联到现有信念或假设的信息，应进入未知证据处理队列。

### 6.5 确认证据

用户在观察池中确认某条观察为证据，并将它关联到一个或多个假设。

确认时需要设置或自动生成：

- 相关性。
- 支持或反对方向。
- 来源可信度。
- 是否立即生成更新预览。

### 6.6 生成更新预览

系统根据证据和假设调用似然模型，生成贝叶斯更新预览。

预览需要展示：

- 当前先验概率。
- 各估计器输出。
- 最终 ensemble 输出。
- 更新后的后验概率。
- 信念强度变化。
- 模型解释。
- 是否建议人工复核。

### 6.7 应用或回滚更新

用户可以应用更新预览。

应用后，系统更新假设当前概率，并生成不可变的更新事件记录。

如果发现自动确认或模型判断错误，用户可以回滚某次更新。回滚应恢复相关假设到更新前快照，并记录回滚时间。

## 7. 权限和访问

- `worldModel` 是独立应用，拥有完整 UI 和 API。
- 用户通过 `myWeb` 管理员后台访问 `/admin/world-model/*`。
- `myWeb` 必须先完成管理员登录校验，再代理请求到 `worldModel`。
- `worldModel` 只接受带内部签名的代理请求。
- 直接访问 `worldModel` 服务且没有有效内部签名时，必须返回未授权。

## 8. 数据和隐私要求

- 世界模型数据存储在独立 Postgres 数据库中。
- 不将世界模型表写入 `myWeb` 的 Prisma schema。
- 采集平台的 cookie、API key、token 等凭据不进入数据库。
- 来源配置只保存凭据引用名。
- `.env`、`.env.local`、密钥文件和真实 token 不得提交到 Git。
- 投资相关信念仅用于个人记录和概率更新，不提供交易执行、公开荐股或自动投资决策。

## 9. 训练数据要求

用户不希望手动标注大量训练数据。

第一版训练数据应尽量来自公开可靠来源，包括：

- FEVER 和 SciFact 等事实核验数据集。
- Metaculus、Polymarket、Manifold 等预测历史。
- GDELT 等事件和新闻数据。
- GH Archive 等 GitHub 事件数据。
- Hugging Face Hub 的模型、数据集和趋势信息。
- Common Crawl 或网页语料中的历史事件信息。

用户自己的已结算假设可以作为后续校准数据，但不是冷启动训练的前提。

## 10. 非目标

第一版不做以下内容：

- 不做公开预测市场。
- 不做多人协作权限体系。
- 不做交易执行。
- 不做公开投资建议页面。
- 不要求一开始覆盖所有社交平台的稳定自动采集。
- 不要求服务器完成所有重模型训练。
- 不把世界模型数据混入个人网站内容数据库。

## 11. 验收标准

第一版完成后，应满足：

- 可以通过 `myWeb` 管理员后台访问完整世界模型 UI。
- 未登录 `myWeb` 管理员账号时无法访问世界模型入口。
- 未带内部签名时无法直接访问 `worldModel`。
- 可以创建信念和假设。
- 可以选择互斥完备或相互独立概率结构。
- 可以手动录入观察。
- 可以配置至少一种外部来源并执行 dry-run 采集。
- 观察进入观察池后可以完成去重检查。
- 可以将观察确认为证据。
- 一条证据可以关联多个假设。
- 可以生成似然模型输出和更新预览。
- 可以应用贝叶斯更新。
- 可以查看更新前后概率变化和模型解释。
- 可以回滚一次更新事件。
- 可以导入或注册一个模型产物。
- 可以运行自动化证据闭环，将来源采集、观察去重、候选证据生成、证据确认和概率更新串联起来。
- 可以编辑证据内容和每个证据-假设关联的方向、相关性、似然比、置信度和解释。
- 可以软删除或拒绝证据；如果证据已经产生更新，系统需要先回滚对应更新，再改变证据状态。
- 可以用图谱查看信念、假设、证据和更新事件之间的影响关系。
- 似然模型训练数据应来自真实公开数据集或本地真实确认证据，不得使用 demo 合成数据。
- 单元测试覆盖贝叶斯更新、似然合成、去重和回滚。
- 构建、类型检查和核心测试命令可以运行并给出明确结果。

## 12. 阶段验收与 Git 保存要求

每一个实施阶段完成后，必须先完成该阶段对应的最小测试、类型检查、构建或浏览器验收，再使用 Git 提交保存阶段成果。

阶段提交要求：

- 提交前运行与本阶段相关的验证命令，并确认失败项已经处理或明确记录。
- 提交前执行 `git status`，确认只包含本阶段相关文件。
- 不提交 `.env`、`.env.local`、API key、token、cookie、密码、模型密钥或其他敏感配置。
- 如果测试、构建或验收未通过，不得提交该阶段成果。
- 如果某项验证因环境缺失无法运行，需要在阶段记录中说明命令、失败原因、是否由本阶段改动引起，以及剩余风险。
- 每个阶段使用独立提交，提交信息应能说明阶段目标，例如 `feat: scaffold world model app`、`feat: add bayesian update core`、`feat: add signed myWeb proxy`。

建议阶段提交点：

1. 项目脚手架和基础验证命令完成后提交。
2. 纯领域逻辑和单元测试通过后提交。
3. Prisma 数据模型、迁移和服务层测试通过后提交。
4. 后台 UI 页面和浏览器验收通过后提交。
5. `myWeb` 同域代理和内部签名校验通过后提交。
6. 观察来源、去重、观察池 dry-run 验收通过后提交。
7. 似然 estimator、模型产物导入和训练脚本验证通过后提交。
8. 全量验收、文档和部署说明完成后提交。

## 13. 下一阶段完善计划

### 13.1 目标

下一阶段目标是把系统从“手动确认证据和表格查看”升级为可运行的本地自动化证据闭环。

系统应能基于已创建的信念和假设，自动生成检索任务，采集观察，识别候选证据，按假设分别评估相关性和似然比，确认或等待人工复核，并在达到阈值时自动应用贝叶斯更新。

同时，系统需要支持对已确认或已回滚证据进行编辑、重新应用、拒绝和软删除，并用图谱展示信念、假设、证据和更新事件之间的动态影响关系。

### 13.2 自动化证据闭环

新增自动化运行能力：

- 用户可以在后台手动触发一次自动化闭环运行。
- CLI 可以通过 `npm run observe -- --loop` 运行一次自动化闭环。
- 自动化闭环输入包括可选信念范围、来源范围、自动应用阈值、单次最大观察数量和运行模式。
- 运行模式包括自动应用和仅生成待审候选两种。

闭环流程：

1. 读取活跃信念和假设。
2. 根据每个信念和假设生成检索查询。
3. 运行已启用来源采集观察。
4. 对观察执行去重。
5. 将观察与相关假设匹配，生成证据候选。
6. 对每个证据-假设关联分别计算方向、相关性、似然比、置信度和解释。
7. 达到自动应用阈值的候选证据自动确认并应用更新。
8. 未达到阈值但有价值的候选证据进入待审队列。
9. 记录本次运行的采集数量、去重数量、候选数量、自动应用数量、待审数量和失败原因。

自动化闭环失败时，系统只记录失败运行和错误原因，不应导致后台页面出现 server-side application error。

### 13.3 证据编辑、删除和重新应用

证据管理需要支持：

- 编辑证据标题、正文、链接、可信度和元数据。
- 编辑每个证据-假设关联的方向、相关性、似然比、置信度和解释。
- 为已有证据新增假设关联。
- 从已有证据移除某个假设关联。
- 拒绝或软删除证据。

证据编辑规则：

- 一条证据可以关联多个假设。
- 同一条证据对不同假设的相关性、方向、似然比和置信度可以不同。
- 已应用证据被编辑后，系统需要先回滚旧更新，再基于新证据关联重新生成并应用更新。
- 已应用证据被拒绝或软删除时，系统需要先回滚对应更新，再把证据状态改为 `REJECTED`。
- 未应用证据被拒绝或软删除时，只改变证据状态，不改变假设概率。
- 已回滚的更新事件不能重复回滚。
- 被拒绝证据不参与后续自动更新和信念强度计算。

删除默认采用软删除，不直接物理删除数据库记录，以保留审计历史和更新事件外键完整性。

### 13.4 真实训练数据管线

训练数据不得使用 demo 合成数据。

v1 不以自训练轻量模型作为主似然模型。训练数据管线的主要目的，是评估和校准 LLM API 评分质量，并为后续蒸馏轻量模型提供真实样本。

第一阶段训练数据来源优先级：

1. 公开真实事实核验和自然语言推理数据集。
2. 本地数据库中用户真实确认过的证据-假设关联。
3. 基于真实历史事件和真实来源构造的样本。

默认候选公开数据集：

- FEVER：事实核验 claim、evidence 和支持/反驳/信息不足标签。
- SciFact：科学 claim、论文证据和蕴含/反驳标签。
- CLIMATE-FEVER：气候相关真实 claim、证据和支持/反驳/信息不足标签。
- CFEVER：中文事实核验数据集，作为中文样本补充候选。

统一训练样本格式：

```json
{
  "source": "fever|scifact|climate_fever|cfever|local_confirmed",
  "claim": "string",
  "evidence": "string",
  "label": "SUPPORTS|OPPOSES|NEUTRAL",
  "relevance": 0.0,
  "likelihoodRatio": 1.0,
  "confidence": 0.0,
  "provenance": {
    "dataset": "string",
    "split": "string",
    "sourceId": "string"
  }
}
```

标签到似然比的默认映射：

- `SUPPORTS` 映射为 `likelihoodRatio = 2.5`。
- `OPPOSES` 映射为 `likelihoodRatio = 0.4`。
- `NEUTRAL` 或信息不足映射为 `likelihoodRatio = 1.0`。

训练脚本需要输出：

- 样本总数。
- 各来源样本数。
- 支持、反对、中性样本分布。
- 训练使用的数据文件路径。
- 模型版本。
- 误差指标。
- 是否成功训练。

LLM 评分评估需要输出：

- LLM 模型名称。
- 评分样本数。
- 支持、反对、中性样本上的方向准确率。
- 似然比分布。
- 低置信度和需要人工复核的比例。
- 与轻量 fallback 输出的分歧比例。

如果外部数据下载失败，训练流程需要明确报错，不能静默生成伪造样本。

### 13.5 信念图谱视图

后台 UI 需要引入图谱视图，用于替代表格作为主要观察方式。

图谱节点：

- 信念节点：显示可读编号、标题和当前强度。
- 假设节点：显示可读编号、命题、当前概率和支持/不支持类型。
- 证据节点：显示可读编号、标题、状态和可信度。
- 更新事件节点：显示可读编号、应用或回滚状态。

图谱边：

- 信念到假设：表示归属关系。
- 证据到假设：表示影响关系，边上显示方向、相关性和似然比。
- 证据到更新事件：表示该证据产生了更新。
- 更新事件到信念：表示更新改变了信念强度。

视觉规则：

- 支持边使用绿色。
- 反对边使用红色。
- 中性或混合边使用灰色。
- 边粗细根据相关性变化。
- 假设节点颜色深浅根据当前概率变化。
- 被拒绝证据和已回滚事件默认置灰。

交互规则：

- 总览页展示全局图谱。
- 信念页展示选中信念的局部图谱。
- 证据页展示证据影响图谱，并保留证据编辑列表。
- 点击节点时展示详情。
- 点击证据节点时可以进入该证据的编辑区域。

图谱实现优先使用成熟 React 图谱库，例如 `@xyflow/react`，避免手写复杂拖拽、缩放和连线逻辑。

### 13.6 测试和验收

服务层测试需要覆盖：

- 自动闭环可以从来源采集观察，生成候选证据，并在达到阈值时自动应用更新。
- 仅待审模式不会改变假设概率。
- 来源失败时生成失败运行记录，而不是抛出页面错误。
- 同一条证据关联多个假设时，每个假设使用独立的相关性和似然比。
- 编辑已应用证据会回滚旧更新，并用新关联重新应用。
- 软删除已应用证据会先回滚，再把证据状态改为 `REJECTED`。
- 软删除未应用证据不会改变假设概率。
- 已回滚事件不能重复回滚。

训练脚本测试需要覆盖：

- 真实公开数据可以转换为统一训练样本格式。
- 每条外部训练样本必须包含 provenance。
- 输出样本中不得出现 `source: "demo"`。
- 标签到似然比的映射符合需求。
- 有真实样本时，可以评估 LLM 主评分器，并可选训练轻量 fallback 模型产物。

UI 和图谱测试需要覆盖：

- 总览页和信念页可以渲染信念、假设、证据和更新事件节点。
- 证据到假设的边能展示方向、相关性和似然比。
- 点击证据节点可以看到证据详情。
- 证据编辑保存后，页面显示新的关联参数和更新状态。
- 证据被拒绝后，图谱节点置灰，相关假设概率恢复到回滚后的值。
